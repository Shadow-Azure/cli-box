# 设计:《我写了一个小工具,让 OpenClaw 更好地使用 Claude Code》

## 1. 背景与目标

写一篇 **实践分享类** markdown 文档,介绍作者当前的一个实践:用自研小工具 cli-box 把 OpenClaw 和 Claude Code 串联,搭出一个"简易但够用"的干活方式。

**目标**:让读者 5–8 分钟读懂"做了什么、解决了什么问题、cli-box 在其中的角色",产生共鸣或启发,而不是被推销。

## 2. 受众与定位

- **读者**:对 AI agent / 自动化工作流感兴趣的开发者
- **定位**:独立实践分享(取向 B),**弱项目宣传**
- **发布位置**:`docs/` 根目录,与 `ai-agent-openclaw-workflow.md` 并列;可外发(博客 / 朋友圈)

## 3. 写作语气约束(关键)

功能仍在完善,本文是**阶段性分享**,不是产品发布。语气必须克制、诚实:

- ✅ 多用:实践、尝试、目前、简易、够用、进行中、体会到
- ❌ 避免:框架、强大、高效、完美、终极、一站式、赋能
- 角色比喻(助理 / 骨干 / 老板)**轻提一次**即可,不贯穿全文
- 收尾须明确"功能仍在完善,本文只分享当前形态与思路"
- "好处"一律写成"目前体会到的好处"

## 4. 文档结构(大纲)

### 0. TL;DR
- 一句话:用小工具 cli-box,让"IM 体验好但易丢任务"的 OpenClaw 去调度"代码强且稳"的 Claude Code;OpenClaw 当助理,Claude Code 当骨干,飞书把进展推回给我。
- 配 **图 1:整体数据流闭环**(Mermaid)
- 定调:"一个进行中的实践"

### 1. 遇到的问题
- OpenClaw(用 MiniMax 模型):飞书 / IM 接入生态和体验好,但跑一段时间任务容易丢,扛不住正经代码开发
- Claude Code:代码能力强、上下文管理稳、不易丢、关掉也能 `-r` 恢复;但 IM 接入 / 体验弱
- 痛点:单用哪一个都不行——没法"在飞书里稳稳地遥控干开发"
- 配:**优劣对比表**(markdown 表格,不用图)

### 2. 怎么解决
- 思路:不二选一,组合——让各自干擅长的事
- 角色分工(轻提一次):OpenClaw = 助理(接消息 / 调度 / 汇报),Claude Code = 骨干(写码 / 跑长任务)
- 缺一个"粘合剂"让助理能操作骨干 → cli-box
- 闭环:`我 → 飞书 → OpenClaw → cli-box → Claude Code → 截图/scrollback → 飞书 → 我`
- **两个并重的好处(本文核心动机)**:
  1. **对我 —— 交互与可见**:准确方便地截图 → 推飞书 → 形成交互闭环;让我**亲眼看到** Claude 的真实运行状态,避免 OpenClaw 转述失真导致错误反馈无法纠正
  2. **对 OpenClaw / 系统 —— 降低门槛**:只需简单 CLI(`list` 管 / `type`·`key`·`click` 操作 / `scrollback`·`screenshot` 查看)就能驱动 Claude Code → **对 OpenClaw 背后模型的代码能力、上下文管理、记忆要求都大幅降低**;重活由 Claude Code 扛(强代码 + 稳上下文 + 可恢复),所以 OpenClaw 用相对弱的模型(minimax)也能当好调度

### 3. cli-box 的机制与好处
- 是什么:macOS 上的一个沙箱小工具,一条命令把任意 CLI 跑在独立窗口,且能被外部程序驱动
- 核心能力(简述,不展开命令细节,配表格):
  - 管:`list`
  - 操作:`type` / `key` / `click`
  - 读:`scrollback`
  - 看:`screenshot`
- 为什么适合当粘合剂:
  - 零侵入(Claude Code 不用做任何适配)
  - 可观测(screenshot + scrollback 让"看"和"读"都可靠)
  - 可控(外部用简单 CLI 就能驱动)
- 目前体会到的好处(对应上面两个动机):
  - **降低门槛**:OpenClaw 不必亲自写码,只需操作 CLI + 看截图/scrollback —— 因此对它的模型代码能力、上下文管理、记忆要求都很低;Claude Code 的强代码 + 稳上下文 + 可恢复被用在该用的地方
  - **可见可控**:我通过飞书拿到准确的视觉反馈,能及时纠正,而不是盲信 OpenClaw 的转述
